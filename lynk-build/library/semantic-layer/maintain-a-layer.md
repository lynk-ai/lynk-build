# Maintain a layer

**What it is** — the fourth arc of [the frame](the-frame.md): what to do *after* a layer ships. Building a layer is a one-time act; keeping it correct is continuous — a layer that passed the build last month can serve wrong answers today because the warehouse moved underneath it, a name was refactored without updating its references, or a metric's meaning drifted (derived: a layer is a set of definitions over a warehouse that changes independently, so a green build is a snapshot, not a guarantee). **Docs v2 has no maintenance-procedure page** — it specs what each primitive must *be* and how a build is validated, not the operational loop for keeping a live layer correct — so the *method* here is this book's own contribution (that is the non-inferable value; the Book Standard · `non-inferable-only`). The **mechanisms** the method drives, by contrast, are the spec's job and are pointed at, never restated: `enabled: false` (Docs v2 `concepts/entity/entity-md.md`, "disables the whole entity"), the reference forms `metric()` / relationship joins / imports (Docs v2 `concepts/entity/schema-yml/{metric,relationships,identity-and-imports}.md`), and the build/validation lifecycle (Docs v2 `concepts/project.md`, "Lifecycle").

**Prerequisites**
- A **shipped, built layer** — one that already compiles at the authoritative surface. Maintenance keeps a green layer green; it is not a first build (→ [build-a-layer](build-a-layer.md)).
- A **read-only warehouse** (the proxy surface for value truth) *and* access to the **compiled Lynk build** (the authoritative surface where fields are consumed) → See the Evals book · `verify-at-the-authoritative-surface`.
- An **anchor** for re-grounding, chosen by the hierarchy in [ground-a-metric](ground-a-metric.md): (1) a curated truth table of verified real values > (2) a sanity invariant > (3) a hand-computed value from raw rows. **Never the candidate's own SQL** — a value that grounds itself proves nothing (source: `.claude/skills/semantic-layer-audit/references/bug-taxonomy.md`, "Anchoring rule").
- A **place to record definition changes** — a `CHANGELOG` file, commit messages, or a supporting doc close to the layer (our practice, not a Docs v2 requirement: it is the Book Standard · `supersede-dont-delete` applied to a metric's meaning, so a corrected value can't quietly regress).

**Steps** (each with one observable outcome)
1. **Rebuild on a schedule, not only when editing.** The build's field-probe is the only thing that surfaces a column that was renamed, retyped, or dropped; an unbuilt layer keeps serving the last good version while its definitions silently rot. → *Outcome:* a scheduled (or on-schema-change) build runs and reports pass or the specific field failures.
2. **On drift, repoint or disable — then re-ground.** When a definition loses its backing column, either repoint it at the new column or set `enabled: false` until it can be repaired (never leave a broken definition live). After any change to a value's `sql` or its columns, re-ground the metric **at the authoritative surface** — a definition that still *compiles* can still compute the wrong number. → *Outcome:* the affected definition is repointed or disabled, and any value change matches its anchor in the compiled build.
3. **Rename as one atomic change, not a local edit.** A `name` is not local: it is reached through several reference forms (`metric()`, relationship joins, imports — the forms are specced in Docs v2 `concepts/entity/schema-yml/{metric,relationships,identity-and-imports}.md`), and the build is the only thing that surfaces a reference the rename missed. The method risk this page owns: treat the rename and every reference update as one change and rebuild before calling it done — a partial rename leaves a dangler that compiles nowhere. → *Outcome:* the rebuild is green with zero dangling references.
4. **Deprecate on a lifecycle, not by delete.** To retire a primitive: (a) set `enabled: false` — which disables the whole entity, per Docs v2 `concepts/entity/entity-md.md` — or drop the single definition, and add a one-line pointer to its replacement; (b) keep it discoverable for one release cycle so consumers and downstream references can migrate; (c) remove only after that cycle, once nothing references it. (The lifecycle is our call — Docs v2 specs `enabled`, not a deprecation policy; delete-in-place is what strands consumers.) → *Outcome:* the retired primitive carries a replacement pointer and survives one cycle before removal.
5. **Record consequential meaning changes in the definition changelog.** When a denominator is corrected, a filter added, or a scale fixed, log *what changed, when, and why*. → *Outcome:* a dated changelog entry exists for the change.

**Verification**
- The scheduled rebuild is **green at the authoritative surface** — every entity emits its CTE and every field resolves as consumed, not merely in raw SQL (a green raw-warehouse check over a red build is false green → See the Evals book · `verify-at-the-authoritative-surface`).
- Every metric touched this cycle **grounds true** against its external anchor in the compiled build (→ [ground-a-metric](ground-a-metric.md)), not against its own SQL.
- No **dangling references** remain — a rename or a removal that missed a `metric()`/`@`/`join_name` reference shows up as a build failure, so a green build is the check.
- The question **"why did this number move since last month?"** is answerable from the definition changelog without a forensic dig.

**Failure modes** (symptom → fix/escape)
- **Never rebuilding** — symptom: consumers get stale-but-plausible answers, no error anywhere, because the layer serves the last good build while a dropped column rots underneath. Fix: schedule the rebuild (step 1); drift is invisible until the build probes it.
- **Repoint without re-grounding** — symptom: the definition compiles again, so it looks fixed, but now computes a *different* wrong number. Fix: re-ground at the authoritative surface after every `sql`/column change (step 2).
- **Grounding from the candidate's own SQL** — symptom: "it matches" because the check ran the very definition it was checking — a circular, false green. Fix: use an external anchor by the [ground-a-metric](ground-a-metric.md) hierarchy.
- **Rename misses a reference** — symptom: `metric() … not found` or a dangling `@` at build time. Fix: find *all* references before renaming; if the loop you exposed is cross-entity, it is a compile cycle → See [break-a-dependency-cycle](break-a-dependency-cycle.md).
- **Hard-delete instead of lifecycle** — symptom: orphaned references fail the build, or consumers are surprised by a definition that vanished with no replacement. Fix: the deprecation lifecycle (step 4) — disable + pointer, one cycle, then remove.
- **No definition changelog** — symptom: an already-fixed value bug silently regresses because nobody re-ran the check that guarded the old definition, and no one can say when the meaning changed. Fix: keep the changelog (step 5); it is what lets you re-run the old value checks.

**Takeaway** — **a shipped layer stays correct only if maintenance is a job: rebuild to catch drift, re-ground every value change at the authoritative surface, rename and deprecate atomically so no reference dangles, and log meaning changes so a fixed bug can't quietly come back.**

**Example** *(constructed, illustrative)* — a `player_game.minutes_played` metric backed by warehouse column `min`. A warehouse migration renames `min` to `minutes`. Nobody edits the layer, so nothing rebuilds — and the layer keeps serving last month's numbers until the next scheduled rebuild probes the field and reports `column "min" does not exist`. The maintainer repoints the metric to `minutes`, re-grounds it against the curated season-totals anchor in the compiled build (it matches), and logs the repoint in the definition changelog. Had they only repointed and skipped the re-ground, a silent unit change (seconds vs minutes) would have shipped compiling-but-wrong.

**In this system** — this is the **maintenance arc** of [the frame](the-frame.md) (the fourth arc, after build / verify / fix): the same build → audit → fix loop, run continuously against a moving warehouse instead of once at construction. It is the Best Context book · `living-sources` applied to a semantic layer — a home outgrows itself and nothing forces the refactor, so someone must own it. The re-grounding it demands is [ground-a-metric](ground-a-metric.md); the cross-entity loop a bad rename can expose is [break-a-dependency-cycle](break-a-dependency-cycle.md); the surface every check must reach is the Evals book · `verify-at-the-authoritative-surface`. For the mechanisms themselves the homes are the real spec pages — `enabled` in Docs v2 `concepts/entity/entity-md.md`, the reference forms in `concepts/entity/schema-yml/{metric,relationships,identity-and-imports}.md`, the build lifecycle in `concepts/project.md` — which this page points at, never restates. There is no maintenance-procedure page in Docs v2 to point at for the *method*: the loop, the sequencing risk, the deprecation lifecycle, and the definition changelog are this book's own contribution, which is why they earn a page here.
