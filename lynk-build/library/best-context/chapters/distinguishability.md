---
name: Distinguishability
description: When a reader must choose between two similar items, the difference must show in BOTH the name and the one-line description — the complement of one-concept-one-home. Read when two similar items must coexist and a reader or model could pick the wrong one — naming metrics, pages, or tools apart.
---

This rule governs two legitimately-different things sitting side by side. A reader — human or agent — choosing between them must be able to tell them apart from their name and their description alone, because that is all a chooser reads before committing. Two items that share a name, or carry identical or near-identical descriptions, force a guess: the choice becomes a coin-flip, and everything built on the wrong pick is silently wrong. It's the complement of [one-concept-one-home](one-concept-one-home.md): one-home says *define each thing once*; distinguishability says *when two distinct things coexist, make the difference visible in both the name and the one-line description*. (Framing ours — one-home kills the duplicate; distinguishability separates the genuine pair.)

A chooser routes on two surfaces, and the difference must live in both. The name fails when two distinct items share it — the reader can't even address them apart — and the fix is to qualify each name with what makes it distinct (`player_total_points` vs `team_total_points`). The description fails when the one-liners are identical or near-identical — the name distinguishes but the description doesn't confirm, or vice versa — and the fix is to state the discriminating fact in each ("…scored by the player…" vs "…by the team…"). Both, not either: fixing only the name leaves two entries whose descriptions still read the same, so a reader who trusts the description is back to guessing; fixing only the description leaves a name collision a tool may not even be able to reference. (Derived: a chooser that reads name-then-description stops at the first surface that resolves the ambiguity, so an ambiguity in *either* surface can strand it.) **Two things a reader must choose between are distinguishable only when the difference shows in BOTH the name and the description — fixing one and leaving the other still leaves a coin-flip.**

A concrete case: in a semantic layer, a `player_game` and a `team_game` entity each defined a metric named `total_points` carrying the identical description "Total points scored across all games." One is a player's total, the other a team's — but an agent (or a text-to-SQL model) selecting a metric by name plus description had no way to tell which was which, so any query that grabbed the wrong one returned player numbers where team numbers were meant, with nothing flagging the error. The fix did both: qualify the names (`player_total_points` / `team_total_points`) *and* disambiguate the descriptions ("…scored by the player…" vs "…by the team…").

Per-page label honesty (a name and description that promise exactly what the body delivers) is necessary but not sufficient: two pages, or two metrics, that each pass in isolation can still be indistinguishable *as a pair* — this rule adds the between-items check no per-page rule can see, and the discriminating fact must be *stated*, not implied, because the chooser cannot infer it. It applies two ways. As an admission test — "would the agent get this wrong without this instruction?" — identical descriptions guarantee it would, so the discriminating fact earns its place. And in semantic-layer authoring, naming metrics distinctly at authoring time avoids the collision, while an audit can flag it (a `distinguishable` check, major severity) when it slips through.
