Every MCP connection (as of spec revision 2025-11-25) starts with a three-step handshake (MCP spec 2025-11-25, Basic → Lifecycle page, for everything in this paragraph and the next): the client sends an `initialize` request carrying `protocolVersion` (e.g. `"2025-11-25"`), its `capabilities`, and `clientInfo`; the server replies with its own capabilities, `serverInfo`, and optional `instructions`; the client then sends a `notifications/initialized` notification. Version mismatch has defined behavior: the server MUST reply with the latest version it supports, and the client SHOULD disconnect if it can't work with that. Over HTTP there is one extra obligation, specified on this same Lifecycle page rather than under transports: after initialization the client MUST send an `MCP-Protocol-Version` header on all subsequent requests.

Capabilities are not decoration — both sides may only use capabilities that were successfully negotiated (MCP spec 2025-11-25, Lifecycle page). A server's declaration looks like `"tools": {"listChanged": true}`, `"resources": {"subscribe": true, "listChanged": true}`, `"logging": {}` in its `initialize` response. The sub-flags: `listChanged` (available on prompts, resources, and tools) signals the server will notify when its list changes; `subscribe` (resources only) signals per-resource update subscriptions. If you use the official SDKs you will likely never write this object: the quickstart servers never declare capabilities manually, so FastMCP/McpServer evidently compute the declaration from what you register (derived — from the quickstarts' omission of any manual declaration; see [build-a-server](build-a-server.md)).

**Transports.** Two are current: **stdio** and **Streamable HTTP**. The history matters because you will still meet the old one in the wild: the original 2024-11-05 revision specified stdio + HTTP+SSE; the **2025-03-26** revision replaced HTTP+SSE with Streamable HTTP (spec PR #206) (both from the 2025-03-26 and 2025-11-25 changelogs, modelcontextprotocol.io). Two transport-specific rules from the same changelog source:

- **stdio**: the JSON-RPC channel is stdout, but servers may log freely to **stderr**. Printing anything else to stdout corrupts the protocol stream (derived — stdout carries the JSON-RPC messages, so non-protocol bytes on it are by construction protocol corruption).
- **Streamable HTTP**: servers must return **HTTP 403** for requests with invalid `Origin` headers — a DNS-rebinding defense the spec makes mandatory rather than advisory.

Choosing between them: stdio is what local, subprocess-launched servers use (it is the default in every quickstart and in `claude mcp add` — see [test-and-connect](test-and-connect.md)); Streamable HTTP is for servers reached over the network. That division of labor is derived from how the official docs and tools consistently pair them, not from a normative rule in the spec.

Note the ground is shifting: the 2026-07-28 release candidate proposes a stateless protocol core that drops this init handshake and the protocol-level session header for remote servers — see [versions-and-governance](versions-and-governance.md) before you hard-code handshake assumptions in long-lived infrastructure.
