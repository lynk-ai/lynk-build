This is the failure that lives *between* windows, not inside one. [context-rot](context-rot.md) degrades an agent as one window fills; cross-session amnesia is different in kind: a task too long for one session spans several, and "each new session begins with no memory of what came before" (Anthropic, *Effective harnesses for long-running agents*, 2025). Compaction is explicitly not the answer — it shrinks one window; it does not carry state across the boundary between windows.

The answer is persistent external state plus a two-phase harness (source: Anthropic, 2025). A one-time initializer agent runs first, leaving behind `init.sh` (to rebuild the environment), a progress log (`claude-progress.txt`), and an initial git commit — the durable state a later session reads. Then each subsequent coding agent works one feature per session: it orients from the progress log and git history, does one unit, commits, updates the log, and the session can end safely. Two design choices make it hold. State is a durable artifact, not a memory — the progress file and git history are the memory, the window is disposable, which is the WRITE lever ([four-operations](four-operations.md)) at session scale. And re-anchor every session — reading the goal back in matters because attention to the original goal decays over turns: the Goal Accessibility Ratio (attention from generated tokens back to system-prompt goal tokens) fell 27–48% within 50 turns across all 10 architectures tested, steepest in the first ten (*When Attention Closes*, Dongre et al., 2026, arXiv 2605.12922 — preprint; the causal ablation is single-model). Restating the goal from durable state is the cheap counter.

**A task that outlives one window needs durable external state, not a bigger window: initialize once, then resume one unit at a time from a progress log and git history — compaction can't cross the boundary.** Anthropic's long-running-agent harness (2025) is the real case: the initializer writes `init.sh`, `claude-progress.txt`, and a first commit; each coding session completes exactly one feature against a machine-readable feature list, commits, and stops — and the one-feature-per-session restriction is what keeps a fresh, amnesiac agent from one-shotting or falsely declaring done. (Caveat: an experimental harness on the Claude Agent SDK, not a shipped product.)

Any agent working a task across days fits this shape: the repository, its git history, and a progress note are the durable state, and each session re-orients from them rather than remembering. See [memory-shapes](memory-shapes.md) for the shape of the state a session leaves (including the tamper-resistant JSON feature list), [context-rot](context-rot.md) for the within-window problem this sits beside, and [four-operations](four-operations.md) — WRITE at session scale.
