Mechanically, a SEP is submitted as a **pull request adding one markdown file to the `seps/` directory** of the spec repo, `modelcontextprotocol/modelcontextprotocol` on GitHub; the file's revision history is the proposal's historical record (modelcontextprotocol.io/community/sep-guidelines; the repo's `seps/` directory). That guidelines page is the authoritative process document, and it describes the live 2026 process: it references the Extensions Track and the SEP-2484 conformance requirement, both of which postdate the December 2025 move to the foundation — so the PR-based flow below is current post-Linux-Foundation (derived from the page's own contents, fetched 2026-07-21; no separate changelog was checked). For what SEPs are, who governs the protocol, and how SEP-932 formalized all this, see [versions-and-governance](versions-and-governance.md) — this chapter is only the *how*.

**Prerequisites.** First the scope gate: the guidelines name what requires a SEP — a new feature or protocol change, any breaking change, a governance/process change, or a complex/controversial topic — and what should skip it: bug fixes, typos, doc clarifications, examples, and non-behavioral schema fixes go as a plain PR instead (SEP guidelines, "When to Write a SEP"). Before drafting, the guidelines direct you to discuss the idea with the relevant Working or Interest Group in Discord — called "the single best way to refine your proposal" — falling back to GitHub Discussions or Discord `#general` if no group covers your area (creating a new group is stated as "still preferable to a cold submission"), and to check alignment with Core Maintainer priorities (the project roadmap) and the design principles, since misaligned proposals "face delays or additional friction" (SEP guidelines, note under "Step-by-Step Process"). Finally, pick the SEP type; there are four — **Standards Track, Informational, Process, and Extensions Track**, the last added for protocol extensions and reviewed identically to Standards Track (SEP guidelines, "SEP Types").

**Steps** (SEP guidelines, "Step-by-Step Process," steps 1–5):

1. Draft the file from `seps/TEMPLATE.md` in the spec repo, named `0000-your-feature-title.md` — `0000` is a deliberate placeholder. The template's preamble header carries **Status, Type, Created, Author(s), Sponsor, PR**; the required sections, in order, are **Abstract (~200 words), Motivation, Specification, Rationale, Backward Compatibility, Security Implications, Reference Implementation**, with Performance Implications, Testing Plan, Alternatives Considered, Open Questions, and Acknowledgments optional (`seps/TEMPLATE.md`; SEP guidelines, "SEP Format"). Don't skimp on Motivation — the guidelines state "SEPs without sufficient motivation may be rejected outright."
2. Open the PR against `modelcontextprotocol/modelcontextprotocol` adding that file to `seps/`.
3. Rename the file to your **PR number** and update the header — PR #1850 becomes `1850-your-feature-title.md`. Live examples in the repo: `414-request-meta.md`, `932-model-context-protocol-governance.md`, `2663-tasks-extension.md` (SEP guidelines; `seps/` directory listing).
4. Find a **Sponsor** — a Core Maintainer or Maintainer, listed in `MAINTAINERS.md` in the spec repo. The documented route: tag 1–2 maintainers whose area relates to your proposal, share the PR in the relevant Discord channel, and if there's no response after **2 weeks**, ask in `#general` (SEP guidelines, "Step-by-Step Process" steps 4–5).
5. Once sponsored, the sponsor assigns themselves to the PR, sets the status to `draft`, and from then on owns every status transition — **authors must not edit the Status field themselves** — and presents the SEP at Core Maintainer meetings (SEP guidelines, "The Sponsor Role," "Status Management"). Your job shifts to revising against feedback; the sponsor moves the status to `in-review` to trigger formal review. Core Maintainers formally review SEPs **every two weeks**, resolving each as `accepted`, `rejected`, or returned for revision (SEP guidelines, "SEP Review & Resolution," steps 7–8).

**Verification** — every stage of this recipe has an observable state. The file sits in `seps/` named by PR number; the canonical status lives in the file's markdown `Status` field with a matching PR label (SEP guidelines, "Status Management"). The named lifecycle statuses are `draft`, `in-review`, `accepted`, `rejected`, `withdrawn`, `final`, `superseded`, and `dormant` (SEP guidelines, "SEP Statuses"). To reach `accepted` you need three explicit things: (1) a **prototype implementation** demonstrating the proposal — an SDK branch, standalone PoC, integration tests, or a reference server/client; the guidelines state pseudocode or a design doc alone is insufficient ("reviewers need to see it"); (2) clear benefit to the ecosystem; (3) community consensus (SEP guidelines, "Prototype Requirements," "SEP Review & Resolution"). The *full* reference implementation is required only before `final`, not before acceptance. For Standards Track SEPs with observable protocol behavior, `final` additionally requires a **conformance scenario merged into `modelcontextprotocol/conformance`** tagged with the SEP number, plus a traceability file `sep-NNNN.yaml` mapping every MUST/MUST NOT and SHOULD/SHOULD NOT in the Specification section to a check ID or a documented exclusion — Process/Informational SEPs and non-observable changes are exempt; the requirement was defined by SEP-2484 (SEP guidelines, "Conformance Test Requirement").

**Failure modes.**

| Symptom | What it means / what to do |
|---|---|
| No sponsor after 6 months | Status becomes `dormant` — explicitly distinct from `rejected`; revivable by finding a sponsor and reopening the PR (SEP guidelines, "SEP Statuses"). |
| Rejected early with little discussion | Likely insufficient Motivation ("may be rejected outright") or misalignment with the roadmap/design principles — both named by the guidelines as friction sources. Return to the pre-submission discussion step. |
| Status label and file disagree, or your status edit got reverted | The markdown `Status` field is canonical and only the sponsor updates it (SEP guidelines, "Status Management"). |
| Prototype pushback despite a detailed design | Expected: a design doc or pseudocode is stated as insufficient for acceptance — ship a runnable PoC or SDK branch (SEP guidelines, "Prototype Requirements"). |
| You find SEPs (932, 1395, 1502) that are GitHub *issues*, not files in `seps/` | A numbering trap, not a second live process: the process was formerly issue-based and early SEPs live as issues numbered by issue number; the current process is PR-based with a checked-in file (derived: those SEPs exist as issues in the spec repo, while the current guidelines mandate PRs into `seps/`). Follow the PR flow. |
