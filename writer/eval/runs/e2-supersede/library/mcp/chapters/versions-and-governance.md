The revision lineage is **2024-11-05 → 2025-03-26 → 2025-06-18 → 2025-11-25**, each spec page naming its predecessor (modelcontextprotocol.io/specification/2025-11-25/changelog; the 2025-06-18 link in the chain is derived from the 2025-11-25 changelog's "since 2025-06-18" header — the 2025-06-18 changelog itself was not fetched, so this book attributes no specific features to that revision). The current stable revision as of this writing (2026-07-21) is **2025-11-25** (modelcontextprotocol.io/specification/latest, fetched 2026-07-21).

What landed where (all from the official changelogs for 2025-03-26 and 2025-11-25):

| Revision | Notable changes |
|---|---|
| 2025-03-26 | Streamable HTTP replaces HTTP+SSE (PR #206); OAuth 2.1 authorization framework; tool annotations (read-only/destructive); JSON-RPC batching; audio content |
| 2025-11-25 | Tasks (experimental); OpenID Connect Discovery for auth-server discovery; icons metadata; sampling gains `tools`/`toolChoice`; JSON Schema 2020-12 as default dialect; `ElicitResult`/enum schema changes + URL-mode elicitation |

One dating trap from the same source: **elicitation predates 2025-11-25** — that changelog only *modifies* elicitation (schemas, URL mode); it did not introduce it. Do not cite 2025-11-25 as elicitation's origin.

**The next revision.** A release candidate for **2026-07-28** is public, with final publication scheduled July 28, 2026 (MCP official blog, blog.modelcontextprotocol.io/posts/2026-07-28-release-candidate/ — headline list below is sourced via a search-result summary of that post, consistent across the official blog and roadmap post, so treat exact details as to-be-confirmed against the final publication). *(superseded 2026-07-21: the RC claims were verified directly against the RC announcement post and the primary spec pages; the search-result-summary caveat no longer applies. The announcement post is dated May 21, 2026, calls this "the largest revision of the protocol since launch," states "The final specification ships on July 28, 2026," and describes a ten-week validation window between RC lock and final publication — blog.modelcontextprotocol.io/posts/2026-07-28-release-candidate/, fetched 2026-07-21.)* Headline changes: a **stateless protocol core** (drops the init handshake and the protocol-level session header for remote servers — a direct break with the lifecycle described in [lifecycle-and-transports](lifecycle-and-transports.md)); an **Extensions framework** (reverse-DNS IDs, negotiated via capability maps, versioned independently in `ext-*` repos); **MCP Apps** (servers ship sandboxed-iframe HTML UIs speaking the same JSON-RPC) *(superseded 2026-07-21: "the same JSON-RPC" overstated it — the iframe UI speaks "its own dialect of MCP, built on JSON-RPC" over postMessage, sharing some methods with core MCP such as `tools/call` but with most methods new under a `ui/` prefix like `ui/initialize` — modelcontextprotocol.io/extensions/apps/overview)*; and **Tasks** moving from experimental core to an extension.

The verified detail, all from the draft-spec changelog (modelcontextprotocol.io/specification/draft/changelog, fetched 2026-07-21) and the extension docs cited inline:

- **Stateless core, confirmed and sharper.** SEP-2575 removes the `initialize`/`notifications/initialized` handshake entirely — protocol version and client capabilities now ride in `_meta` on every request (`io.modelcontextprotocol/protocolVersion`, `io.modelcontextprotocol/clientCapabilities`, plus SHOULD-level `clientInfo`/`serverInfo`), with mismatches returning `UnsupportedProtocolVersionError`. Separately, SEP-2567 removes protocol-level sessions and the `Mcp-Session-Id` header from Streamable HTTP; cross-call state becomes explicit server-minted handles passed as ordinary tool arguments. A mandatory new RPC, `server/discover`, accompanies this: servers MUST implement it to advertise supported versions/capabilities/identity, and clients MAY call it up front or use it as a backward-compat probe on STDIO. The break with [lifecycle-and-transports](lifecycle-and-transports.md) stands — that chapter's handshake and session model describe 2025-11-25 and earlier only.
- **Extensions framework, confirmed with one precision fix.** IDs are `{vendor-prefix}/{extension-name}`, where third parties use a reverse-DNS prefix they own (`com.example/my-extension`) but *official* extensions use the `io.modelcontextprotocol` prefix (e.g. `io.modelcontextprotocol/ui`) — the earlier "reverse-DNS IDs" phrasing covered only the third-party half. Negotiation is via a new `extensions` field in ClientCapabilities/ServerCapabilities (each extension defines its own settings-object schema); official specs live in `ext-*` repos in the MCP GitHub org (`ext-auth`, `ext-apps`), with `experimental-ext-*` for incubation, and extensions evolve independently of the core protocol — extension-repo maintainers ship updates without core-maintainer review, and breaking changes take a new identifier (e.g. `…/my-extension-v2`) (modelcontextprotocol.io/docs/extensions/overview, SEP-2133; draft changelog). One reading trap from that docs page: its negotiation examples still show the old `initialize`-based capability exchange with `protocolVersion: 2025-06-18` — the page predates the RC's handshake removal; do not read it as evidence that `initialize` survives in 2026-07-28.
- **MCP Apps, corrected mechanism.** Tool descriptions carry `_meta.ui.resourceUri` pointing at a `ui://` resource (mime `text/html;profile=mcp-app`) the host can preload; the HTML runs in a host-controlled sandboxed iframe and talks to the host over postMessage in the `ui/`-prefixed MCP dialect described above; `_meta.ui.csp` and `permissions` govern external origins and extra capabilities. The extension is `io.modelcontextprotocol/ui`, repo `modelcontextprotocol/ext-apps`, with its own spec versioned 2026-01-26 (modelcontextprotocol.io/extensions/apps/overview).
- **Tasks → extension, confirmed with mechanism detail.** SEP-2663 moves experimental tasks into the official extension `io.modelcontextprotocol/tasks`; the redesign replaces the blocking `tasks/result` with polling via `tasks/get`, adds `tasks/update` for client-to-server mid-flight input, removes `tasks/list`, and lets servers return task handles unsolicited — no per-request opt-in (draft changelog; docs/extensions/overview).

Headline changes the earlier summary missed (same draft changelog):

- **Multi Round-Trip Requests (MRTR, SEP-2322)** replace *all* server-initiated requests (`roots/list`, `sampling/createMessage`, `elicitation/create`): servers return an `InputRequiredResult` (`resultType: "input_required"`) carrying `inputRequests`, and clients retry the original request with `inputResponses` plus server-opaque `requestState`. Every result now carries a required `resultType` field.
- **Subscriptions rework**: a single long-lived POST-response stream, `subscriptions/listen`, replaces the HTTP GET endpoint and `resources/subscribe`/`unsubscribe`.
- **SSE resumability removed** (`Last-Event-ID`, event IDs): a broken stream loses the in-flight request and clients MUST re-issue with a new request ID.
- **Removals**: `ping`, `logging/setLevel` (log level now per-request via `_meta`), and `notifications/roots/list_changed`.
- **Formal deprecations under a new feature-lifecycle policy** (SEP-2596: Active/Deprecated/Removed states, minimum twelve-month deprecation window, public deprecated-features registry): Roots, Sampling, and Logging are Deprecated (SEP-2577), and OAuth Dynamic Client Registration (RFC 7591) is deprecated in favor of Client ID Metadata Documents.
- Smaller but load-bearing: required Streamable-HTTP headers `Mcp-Method`/`Mcp-Name` (SEP-2243); required `ttlMs` + `cacheScope` caching fields on list/read results via a `CacheableResult` interface (SEP-2549); tool schemas loosened to full JSON Schema 2020-12 (SEP-2106); RFC 9207 `iss` validation mandatory when present (SEP-2468); error codes `-32020`…`-32099` reserved for the spec.

Two practical notes. First, a URL trap: until July 28 the RC spec content is still published under the `draft` path — `modelcontextprotocol.io/specification/2026-07-28/changelog` returned 404 while `modelcontextprotocol.io/specification/draft/changelog` (self-described as changes "since the previous revision, 2025-11-25") is the authoritative changelog (direct fetch attempts, 2026-07-21). Cite `/draft/` URLs for RC material until final publication. Second, beta SDKs implementing the RC shipped June 29, 2026 ("Beta SDKs for the 2026-07-28 MCP Spec Release Candidate Are Here"), and a June 18, 2026 post declared the Enterprise-Managed Authorization extension stable — both per the official blog index's post titles and dates only; the posts themselves were not read, so treat details beyond title and date as unverified.

**Governance.** MCP adopted a formal governance model in **July 2025**: SEPs — Specification Enhancement Proposals — as the change mechanism (formalized as SEP-932 in the 2025-11-25 revision), plus Working and Interest Groups (blog.modelcontextprotocol.io, "Building to Last", 2025-07-31; 2025-11-25 changelog). On **December 9, 2025**, Anthropic donated MCP to the newly created **Agentic AI Foundation**, a directed fund under the **Linux Foundation**, as a founding project alongside Block's goose and OpenAI's AGENTS.md; the steering committee draws from Anthropic, OpenAI, Microsoft, Google, and Amazon, and day-to-day maintainer governance stayed unchanged (blog.modelcontextprotocol.io "MCP joins the Agentic AI Foundation", 2025-12-09; Anthropic's donation announcement). The practical consequence (derived): "will Anthropic change this under us?" is no longer the right risk question — protocol changes go through SEPs under a multi-vendor foundation.
