MCP has two current transports — stdio for local child-process servers, Streamable HTTP for everything remote — plus one deprecated ancestor you'll still meet in the wild. All rules below are from the transports page (modelcontextprotocol.io/specification/2025-11-25/basic/transports) unless noted.

**stdio.** Messages are newline-delimited JSON-RPC, UTF-8, with no embedded newlines. The hard rule, quoted from the spec: "the server MUST NOT write anything to its stdout that is not a valid MCP message." stderr, by contrast, MAY be used for any logging — informational included, a 2025-11-25 clarification (PR #670) — and clients SHOULD NOT treat stderr output as an error signal. Clients SHOULD support stdio whenever possible. Our call: the stdout rule is the most common way new servers break; the per-language specifics live in [build-a-server](build-a-server.md).

**Streamable HTTP.** One single MCP endpoint supports both POST and GET. Every client JSON-RPC message is a new POST carrying `Accept: application/json, text/event-stream`; the server answers a request with either a single JSON body or an SSE stream (its choice). A client GET opens a server→client SSE stream for unsolicited messages — or gets 405 if the server doesn't offer one. Notifications and responses POSTed by the client are acknowledged with `202 Accepted`. Derived reading tip: "Streamable HTTP" is not "SSE transport" — SSE is an optional response mode inside it, which is why a server returning plain JSON bodies is still fully conformant.

Sessions are server-optional: the server MAY mint an `MCP-Session-Id` header on the InitializeResult (visible-ASCII only, cryptographically secure). Once minted, clients MUST echo it on every request; a request missing the ID gets 400; an expired session gets 404, after which the client MUST re-initialize. Explicit termination is HTTP DELETE to the endpoint. So the status codes are a diagnostic table (derived from those rules): 400 without a session header means "you forgot to echo the ID," 404 means "your session died — start over," and neither means the server is down.

Resumability, for dropped SSE streams: servers may attach `id`s to SSE events; a client reconnecting sends `Last-Event-ID` on its GET and the server replays from there — but cursors are per-stream, with no cross-stream replay. Derived consequence: resumption recovers *that stream's* missed events only; anything sent on other streams while you were gone is not replayed.

Security is part of the transport contract, not an add-on: servers MUST validate the `Origin` header and respond 403 to invalid origins — a DNS-rebinding defense made explicit in 2025-11-25 (PR #1439) — and local servers SHOULD bind to 127.0.0.1 rather than 0.0.0.0 (transports page, Security Warning). Why this matters and what happened when tools skipped it is in [security](security.md).

**The deprecated ancestor.** The original HTTP transport (2024-11-05) was **HTTP+SSE**: two endpoints — an SSE endpoint that emits an `endpoint` event telling the client where to POST, plus that POST endpoint. Streamable HTTP replaced it in 2025-03-26 (changelog, PR #206). Old servers persist, so the spec documents a back-compat probe (transports page, backwards-compatibility section): POST an InitializeRequest to the URL; on 4xx (400/404/405), GET the same URL and expect an `endpoint` SSE event — if it arrives, you're talking to a legacy server and should fall back to the old flow. Derived flip side: a modern server receiving a bare GET first (no session, no prior POST) is being probed by a legacy-aware client, which is normal, not an attack.
