The cleanest way to keep MCP's six primitives straight is by **who controls each**. **Tools** are "model-controlled" (that word is verbatim on the tools page, modelcontextprotocol.io/specification/2025-11-25/server/tools): the LLM decides when to call them. **Resources** are application-controlled context — URI-addressed data the host chooses to attach, with optional change subscription — and **prompts** are user-controlled templates, typically surfaced as slash-commands. The resources/prompts characterizations and their method-name strings (`resources/list` / `resources/read` / `subscribe`; `prompts/list` / `prompts/get`) are **unverified**: this book's research pass fetched the tools page but not the spec's resources or prompts pages, so check those pages before relying on the exact strings. The control model is the design guidance (our call, resting on the tools page's verbatim "model-controlled" plus the unverified-but-conventional reading of the other two): if the *model* should decide, make it a tool; if the *app* curates it into context, a resource; if the *user* invokes it deliberately, a prompt.

The tools protocol is two methods — `tools/list` (paginated via `cursor`/`nextCursor`) and `tools/call` — and a tool definition carries: `name`, optional `title`, `description`, `inputSchema`, optional `outputSchema` + `structuredContent`, optional `annotations`, and optional `execution.taskSupport` (tools page). The contract details that actually bite:

- `inputSchema` must be a valid JSON Schema **object, never null**; the default dialect is JSON Schema 2020-12 (tools page). For a parameterless tool the spec's recommended schema is `{ "type": "object", "additionalProperties": false }` (tools page, Tool section).
- Tool names were only formalized in 2025-11-25 (SEP-986): 1–128 chars, case-sensitive, only `A-Za-z0-9_-.`, unique per server (tools page + changelog). Derived consequence: names that older clients tolerated — spaces, slashes, unicode — are now spec violations.
- **Two error channels exist and they are not interchangeable** (tools page): an execution failure ("city not found") returns a normal result with `isError: true` inside it, so the *model* sees the failure and can self-correct; a protocol error (e.g. -32602 "Unknown tool") is a JSON-RPC error the model typically never sees. Returning protocol errors for business failures robs the model of its chance to retry — that framing is derived from the two channels' definitions.

Tool metadata is attack surface, per the spec's own warning blocks (tools page): "clients MUST consider tool annotations to be untrusted unless they come from trusted servers," and clients SHOULD keep a human in the loop with the ability to deny tool invocations. The full exploitation story is in [security](security.md).

The three client-side primitives, briefly — servers request, clients decide. **Sampling**: the server asks the client's LLM for a completion via `sampling/createMessage`; the client keeps control of model choice and user approval, and 2025-11-25 added tool use inside sampling. **Roots**: the client exposes filesystem boundaries via `roots/list` plus `notifications/roots/list_changed`. **Elicitation** (added 2025-06-18): the server requests structured user input mid-flow via `elicitation/create` with a JSON-schema form, plus a URL mode added in 2025-11-25. Semantics and dates are sourced from the spec's client pages and changelogs; the three method-name strings are derived from those pages and the schema rather than quoted, so verify against the schema before hard-coding them. All three are negotiated capabilities — never assume a client supports them (see [protocol-lifecycle-and-versioning](protocol-lifecycle-and-versioning.md)).
