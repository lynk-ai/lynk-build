Every MCP session runs exactly three phases ã initialization, operation, shutdown ã per the lifecycle page (modelcontextprotocol.io/specification/2025-11-25/basic/lifecycle). Initialization is a fixed triple: the client sends an `initialize` request, the server responds, and the client sends `notifications/initialized`. Until that completes, traffic is restricted: the client may send only pings; the server only pings and logging (same source). A server that answers `tools/list` before `initialized` ã or a client that sends it ã is out of spec, and this restriction is a common source of "works in one client, hangs in another" startup bugs (that last observation is our call, derived from the restriction itself: clients differ in how strictly they enforce it).

Two negotiations ride inside `initialize`. **Version**: MCP versions are `YYYY-MM-DD` strings marking "the last date backwards-incompatible changes were made" (versioning page, quoted). The client sends its latest supported `protocolVersion`; the server echoes it if supported, otherwise replies with its own latest; if the client can't accept that answer it SHOULD disconnect (lifecycle page). Over HTTP there is a second step easy to miss: after initialization, every request MUST carry an `MCP-Protocol-Version` header; a server that receives none SHOULD assume `2025-03-26`, and an invalid version gets 400 Bad Request (transports page, Protocol Version Header section). Derived trap: a client that negotiates 2025-11-25 in the handshake but omits the header afterward is silently downgraded to 2025-03-26 semantics on some servers.

**Capabilities**: both sides declare what they support, and MUST only use what was negotiated (lifecycle page, capability-negotiation section). The spec enumerates the keys ã client: `roots` (with `listChanged`), `sampling`, `elicitation`, `tasks`, `experimental`; server: `prompts`, `resources` (with `subscribe` + `listChanged`), `tools` (with `listChanged`), `logging`, `completions`, `tasks`, `experimental` (same source). Derived consequence: a server can never assume its client supports sampling or elicitation ã check the negotiated capabilities or degrade gracefully.

Shutdown is deliberately unceremonious: **no shutdown message exists** (lifecycle page). For stdio, the client closes the server's stdin, waits, then SIGTERM, then SIGKILL; for HTTP, shutdown is just closing connections (same source). If you're looking for a `shutdown` RPC to hook cleanup logic onto, stop ã put cleanup in your process's signal/exit handlers instead (derived from the above).

The batching row is the one to remember when reading old code or blog posts: anything built around JSON-RPC batches targets a spec window of exactly one revision (derived from the changelog rows above).

Version-lifecycle rules that matter for planning: revisions are marked Draft / Current / Final, and deprecated features stay in the spec at least 12 months (or at least 90 days under expedited removal) before removal (versioning page). Looking forward: a **2026-07-28 release candidate** exists ã RC locked May 21, 2026, final targeted July 28, 2026, with a ten-week migration window ã and it is a major break: stateless core (drops the `initialize` handshake and `MCP-Session-Id`), Tasks reshaped as an opt-in extension, roots/sampling/logging formally deprecated, the missing-resource error moving -32002 ã -32602, extensions with reverse-DNS IDs, MCP Apps (server-rendered UI in sandboxed iframes), and W3C trace-context propagation (blog.modelcontextprotocol.io RC post). As of this writing (2026-07-21, one week before the target date) that revision is **still an RC, not final** ã build against 2025-11-25 and treat everything in this paragraph as subject to change until the final ships.
